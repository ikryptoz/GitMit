# Security Policy

## Reporting a Vulnerability

If you discover a security issue, please do not open a public issue first.

Report privately with:
- affected component,
- reproduction steps,
- impact assessment,
- optional proof-of-concept.

Preferred channel:
- create a private security report in GitHub Security tab (if enabled), or
- contact project maintainers through private channel.

## Scope

Security reports are especially welcome for:
- encryption/key handling,
- authentication and authorization,
- sensitive data exposure,
- backend rule bypasses,
- dependency supply-chain risks.

## Response Expectations

Maintainers aim to:
- acknowledge receipt,
- validate and triage severity,
- prepare remediation,
- coordinate responsible disclosure.

## Hardening Principles

- never commit secrets,
- keep dependencies updated,
- use least privilege for tokens/keys,
- keep critical trust decisions server-side,
- document security limitations transparently.
